# PAGE_FLOW.md

## 1. 文件目的

本文件用于明确《哄她模拟器》MVP 的页面流程。

重点回答：

1. 用户从哪里进入；
2. 每一步点击后去哪；
3. 哪些页面允许游客使用；
4. 哪些行为需要登录；
5. MVP 第一版只做哪些页面；
6. 哪些页面暂不做。

---

## 2. MVP 页面列表

| 路由 | 页面名称 | 核心作用 | 是否需要登录 |
|---|---|---|---|
| `/` | 首页 | 展示产品定位，提供训练入口和救急入口 | 否 |
| `/characters` | 角色选择页 | 选择 5 个女友角色之一 | 否 |
| `/characters/[characterType]/levels` | 关卡选择页 | 展示该角色下的 3 个关卡 | 否 |
| `/training/[levelKey]` | 三轮训练页 | 用户进行三轮对话训练 | 否 |
| `/training/[resultId]/result` | 评分复盘页 | 展示评分、等级、复盘建议；游客阶段 `resultId = guestSessionId`，登录保存后可切换为正式 `sessionId` | 否 |
| `/emergency` | 救急分析页 | 用户输入真实聊天，获得即时分析 | 否 |
| `/login` | 登录页 | 用户登录 | 否 |
| `/register` | 注册页 | 用户注册 | 否 |
| `/history` | 历史记录页 | 查看已保存训练记录 | 是 |

## 3. 主流程一：模拟训练流程

```txt
首页 /
└── 点击「开始模拟训练」
    └── 角色选择 /characters
        └── 选择一个女友角色
            └── 关卡选择 /characters/[characterType]/levels
                └── 选择一个关卡
                    └── 三轮训练 /training/[levelKey]
                        ├── 第 1 轮：女友发起情绪表达，用户回复
                        ├── 第 2 轮：女友根据回复继续反应，用户回复
                        ├── 第 3 轮：用户尝试修复关系
                        └── 评分复盘 /training/[resultId]/result
                            ├── 查看总分
                            ├── 查看情绪值 / 信任值变化
                            ├── 查看踩雷点
                            ├── 查看更优回复
                            ├── 继续下一关
                            ├── 换角色
                            └── 登录保存
```

---

## 4. 主流程二：救急模式流程

```txt
首页 /
└── 点击「真实聊天救急」
    └── 救急分析 /emergency
        ├── 用户输入女朋友发来的话
        ├── 系统分析情绪
        ├── 系统分析潜台词
        ├── 系统提示当前雷区
        ├── 系统给出建议回复方向
        ├── 系统给出示例回复
        ├── 用户可复制建议回复
        ├── 用户可选择登录保存
        └── 转为模拟训练（后续增强功能，MVP 可不做）
```

## 5. 游客登录保存流程

游客可以完成训练，但训练数据先存在本地 `localStorage`。

游客阶段统一使用：

1. `guestSessionId`：游客临时训练结果 ID；
2. `levelKey`：业务关卡 ID；
3. `localStorage`：保存整条游客训练结果。

```txt
游客完成训练
└── 生成 guestSessionId
    └── 写入 localStorage
        └── 进入评分复盘页 /training/[guestSessionId]/result
            └── 点击「登录保存」
                └── 跳转登录页 /login
                    └── 登录成功
                        └── 回到评分复盘页
                            └── 调用 mergeGuestProgress
                                ├── 读取 localStorage 中的训练数据
                                ├── 使用 guestSessionId 定位游客结果
                                ├── 使用 levelKey 查询正式 levels.id
                                ├── 写入数据库
                                ├── 绑定当前 userId
                                ├── 生成正式 sessionId
                                ├── 清除 localStorage
                                └── 显示「已保存到你的训练记录」
```

## 6. 登录状态判断规则

| 场景 | 游客 | 登录用户 |
|---|---|---|
| 进入首页 | 可以 | 可以 |
| 进入角色选择页 | 可以 | 可以 |
| 进入关卡选择页 | 可以 | 可以 |
| 完成三轮训练 | 可以 | 可以 |
| 查看当次复盘 | 可以 | 可以 |
| 保存训练记录 | 需要登录 | 可以 |
| 查看历史记录 | 不可以，跳转登录 | 可以 |
| 使用救急分析 | 可以 | 可以 |
| 保存救急分析 | 需要登录 + 主动确认 | 可以，但必须主动确认 |

## 7. 各页面核心功能

### 7.1 首页 `/`

核心功能：

1. 展示产品一句话介绍；
2. 展示核心价值观；
3. 提供两个主要入口：
   - 模拟训练；
   - 真实聊天救急。

页面重点：

> 让用户快速知道这是一个恋爱沟通训练产品，不是普通聊天机器人。

建议首页主文案：

> 不是教你骗她原谅，而是教你真正听懂她。

主要按钮：

1. 开始模拟训练；
2. 真实聊天救急。

---

### 7.2 角色选择页 `/characters`

核心功能：

1. 展示 5 个女友角色；
2. 每个角色显示：
   - 角色类型；
   - 核心难题；
   - 训练能力；
   - 简短介绍。
3. 用户点击角色进入关卡选择页。

角色包括：

1. 缺安全感型；
2. 高期待型；
3. 理性独立型；
4. 情绪外放型；
5. 冷淡压抑型。

页面目标：

> 让用户知道不同角色代表不同沟通难题。

---

### 7.3 关卡选择页 `/characters/[characterType]/levels`

核心功能：

1. 展示当前角色下的 3 个关卡；
2. 每个关卡显示：
   - 场景名称；
   - 简短背景；
   - 训练目标；
   - 难度；
   - 是否已完成。
3. 用户点击关卡进入训练页。

页面目标：

> 让用户明确这次训练要解决什么沟通问题。

---

### 7.4 三轮训练页 `/training/[levelKey]`

核心功能：

1. 展示关卡背景；
2. 展示本关任务目标；
3. 展示当前轮次；
4. 展示当前情绪值；
5. 展示当前信任值；
6. 展示女友当前发言；
7. 用户输入回复；
8. 用户可查看参考回复；
9. 提交后进入 AI 双 Step 流程：
   - Girlfriend Response Agent；
   - Judge Scoring Agent。
10. 完成 3 轮后进入评分复盘页。

重要规则：

1. 无关输入不推进剧情；
2. 低质量输入可以推进，但评分较低；
3. 有害输入需要风险提醒；
4. AI 输出必须经过 Zod Schema 校验；
5. 页面不能因为 AI 输出异常而崩溃。

页面目标：

> 让用户完成一次可控的三轮恋爱沟通训练。

---

### 7.5 评分复盘页 `/training/[resultId]/result`

核心功能：

1. 展示总分；
2. 展示等级；
3. 展示结局类型；
4. 展示情绪值变化；
5. 展示信任值变化；
6. 展示多维评分；
7. 展示本关踩雷点；
8. 展示更优回复；
9. 展示沟通原则；
10. 提供下一步按钮：
    - 继续下一关；
    - 换角色；
    - 登录保存；
    - 查看历史记录。

游客状态：

1. 可以看当次复盘；
2. 结果页此时使用 `guestSessionId` 作为临时结果 ID；
3. 点击登录保存后触发游客数据合并；
4. 合并成功后生成正式 `sessionId`，供后续历史记录和登录态结果页使用。

登录状态：

1. 可直接保存训练记录；
2. 可进入历史记录页。

页面目标：

> 让用户知道自己哪里做得好、哪里踩雷、下次应该怎么说。

---

### 7.6 救急分析页 `/emergency`

核心功能：

1. 用户输入女朋友发来的话；
2. 系统即时分析：
   - 可能情绪；
   - 潜台词；
   - 当前雷区；
   - 回复方向；
   - 示例回复。
3. 默认不保存真实聊天；
4. 用户主动勾选后才保存；
5. 后续可扩展“转为模拟训练”。

隐私规则：

> 不勾选保存，就不写入数据库。

页面目标：

> 先解决用户当下不知道怎么回的问题，再引导他进入训练。

---

### 7.7 登录页 `/login`

核心功能：

1. 用户登录；
2. 登录成功后根据来源返回：
   - 如果来自评分复盘页，返回结果页并保存游客数据；
   - 如果来自历史记录页，返回历史记录页；
   - 默认返回首页。

页面目标：

> 让用户顺利登录，并尽量不中断原来的使用流程。

---

### 7.8 注册页 `/register`

核心功能：

1. 用户注册；
2. 注册成功后自动登录或跳转登录页；
3. 如果用户来自“登录保存”，注册后应继续完成保存流程。

页面目标：

> 让新用户注册后能继续刚才的训练保存流程。

---

### 7.9 历史记录页 `/history`

核心功能：

1. 查看已保存训练记录；
2. 展示每次训练：
   - 角色；
   - 关卡；
   - 得分；
   - 等级；
   - 结局；
   - 完成时间。
3. 点击某条记录查看复盘详情。

访问规则：

1. 需要登录；
2. 未登录访问时跳转 `/login`；
3. 用户只能查看自己的训练记录。

页面目标：

> 让用户看到自己的训练历史，为后续成长报告做基础。

---

## 8. MVP 暂不做页面

MVP 第一版暂不做：

| 页面 | 是否做 | 原因 |
|---|---|---|
| `/progress` 成长报告页 | 暂不做 | 可后续做 |
| `/settings` 设置页 | 暂不做 | 可后续做 |
| `/admin` 管理后台 | 不做 | MVP 不需要 |
| `/leaderboard` 排行榜 | 不做 | 与单人训练定位不符 |
| `/community` 社区 | 不做 | MVP 不做社交 |
| `/pricing` 支付页 | 不做 | MVP 不做商业化 |
| `/chat` 无限聊天页 | 不做 | 违反 MVP 边界 |
| `/ocr` 截图识别页 | 不做 | MVP 不做 OCR |
| `/import` 微信导入页 | 不做 | MVP 不做聊天记录导入 |

## 9. 页面之间的关键跳转

| 起点 | 用户动作 | 终点 |
|---|---|---|
| `/` | 点击开始模拟训练 | `/characters` |
| `/` | 点击真实聊天救急 | `/emergency` |
| `/characters` | 选择角色 | `/characters/[characterType]/levels` |
| `/characters/[characterType]/levels` | 选择关卡 | `/training/[levelKey]` |
| `/training/[levelKey]` | 完成三轮训练 | 游客：`/training/[guestSessionId]/result`；已保存登录结果：`/training/[sessionId]/result` |
| `/training/[resultId]/result` | 点击继续下一关 | 下一个 `/training/[levelKey]` |
| `/training/[resultId]/result` | 点击换角色 | `/characters` |
| `/training/[resultId]/result` | 点击登录保存 | `/login` |
| `/login` | 登录成功 | 返回来源页 |
| `/register` | 注册成功 | 返回来源页或首页 |
| `/history` | 未登录访问 | `/login` |

## 10. 第一阶段开发原则

第一阶段只需要先做页面骨架和流程跳转。

暂时不要做：

1. 数据库；
2. BetterAuth；
3. 真实 AI；
4. 复杂评分；
5. 救急转训练；
6. 成长报告；
7. 管理后台。

第一阶段目标：

> 让用户能用 mock 数据从首页走完整个训练流程。

---

## 11. 第一阶段验收标准

第一阶段页面流程至少满足：

1. 用户能从首页进入角色选择；
2. 用户能选择角色；
3. 用户能选择关卡；
4. 用户能进入训练页；
5. 用户能用 mock 数据完成三轮训练；
6. 用户能看到 mock 评分复盘；
7. 用户能从复盘页返回角色选择；
8. 用户能从首页进入救急页；
9. 未登录访问历史页时，有登录提示或跳转逻辑；
10. 页面结构为后续接入 AI、数据库、BetterAuth 预留位置。

---

## 12. 给 Codex 的开发提醒

执行页面流程开发时，请遵守：

1. 先阅读 `AGENTS.md` 和 `docs/SPEC.md`；
2. 本阶段只做页面骨架和 mock 流程；
3. 不接入真实 AI；
4. 不接入数据库；
5. 不接入 BetterAuth；
6. 不扩展 SPEC 之外的功能；
7. 页面结构要方便后续接入真实数据；
8. 代码保持清晰，适合后续迭代。
